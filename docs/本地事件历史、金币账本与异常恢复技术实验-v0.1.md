# 代办：本地事件历史、金币账本与异常恢复技术实验 v0.1

> 状态：第二个技术小实验已完成首轮实现与自动验证  
> 更新时间：2026-07-18  
> 性质：阶段 0 风险验证，不等于阶段 1 正式产品

## 1. 目标与结论

这个实验只回答五个高风险问题：

1. 任务当前状态、历史事件和金币交易能否在本机长期保存。
2. 完成或撤销能否整笔成功或整笔失败，不留下半条状态。
3. 重复点击、响应丢失后重试和并发操作能否避免重复发币。
4. 进程在提交前被强制终止后，数据库能否重开并恢复一致状态。
5. 周报能否从历史事实中查询“仍然有效的完成”，而不是读取易变的界面数组。

当前结论：下方验收矩阵覆盖的风险均已通过。实现采用 SQLite 本地库、只追加任务事件、只追加奖励交易和幂等命令回执；创建、未完成标题修改、待办可选期限、任意待办完成、历史保留型删除、队列重排和撤销完成已经接入桌面面板。标题修改新增 `title_updated` 并把数据库升级到 schema v3；可选期限新增可空 `deadlineOn` 投影和 `deadline_updated`，当前源码 schema 为 v4。v1、v2、v3 文件可按级迁移到 v4，磁盘旧库升级前会生成并验证 before-v4 一致性备份。删除继续复用已有的 `abandoned` 类型。延期、阻塞、带原因的不再做/重新打开、喂食、通用自动备份、导出和恢复仍未实现。

## 2. 为什么选这条方案

### 2.1 依赖选择

本轮采用：

```toml
rusqlite = { version = "0.40.1", default-features = false, features = ["bundled"] }
uuid = { version = "1.23.5", features = ["v4"] }
```

- `rusqlite` 解决连接、参数绑定、事务和 SQLite 错误处理，替代手写 SQLite FFI。
- `bundled` 把确定版本的 SQLite 编进应用，避免 Windows 机器缺少或误加载 `sqlite3.dll`。
- `uuid` 为任务、事件和交易生成跨进程、跨重启稳定的本地标识。
- 代价是 Rust 原生构建增加 SQLite 编译时间和二进制体积；每个应用实例只持有一个连接，预期负载仍是单机、低并发和短事务，但适配层不假设永远只有一个实例，会通过 SQLite 事务处理跨连接竞争。当前不需要 SQLx 的异步连接池、查询宏和更重的运行时。

第三方库被隔离在 SQLite 适配层。领域模型、状态转换和 Tauri 命令都不暴露 `Connection`、`Transaction`、SQL 列名或 `rusqlite::Error`。

### 2.2 唯一事实来源

```text
桌面 ES modules
        ├─ state / selectors / views（状态与投影）
        ├─ ledger-contract / ledger-session（协议与账本应用服务）
        └─ infrastructure（Tauri 网关与恢复操作箱）
        ↓ 只发送用户意图和 operationId
Tauri 薄命令（ledger/commands.rs）
        ↓
TaskService 应用服务
        ↓ 纯领域转换
Task / TaskEvent / RewardTransaction
        ↓ 单次 commitTransition
SqliteLedgerStore
```

原网页原型中的内存数组和金币计数器没有复用到正式链路。桌面 JavaScript 只渲染 Rust 返回的快照，不在前端复制任务状态机或奖励规则。

## 3. 第一版数据边界

### 3.1 `Task` 当前投影

至少保存稳定 ID、标题、可空 `deadlineOn` 日历日、状态、队列位置、延期时间、阻塞/放弃原因、当前有效完成事件、版本和创建/更新时间。

本轮真正开放的写入状态只有：

```text
新建 → pending
任意立即可执行的 pending → completed
任意立即可执行的 pending → 删除 → abandoned
pending 队列 → 调整顺序 → pending 队列
pending → 修改标题 → pending（顺序不变）
pending → 设置 / 修改 / 清除期限 → pending（顺序不变）
completed → 撤销完成 → pending 队尾
```

数据库结构已经为 `blocked`、`abandoned` 和 `deferUntil` 保留字段，并已实现独立的可空 `deadlineOn`。当前只有删除使用 `abandoned`：任务投影保留并移出队列，`abandonReason=用户删除`；带用户原因的“不再做”、重新打开、阻塞和延期命令仍未实现。`deadlineOn` 不代替 `deferUntil`，不决定任务是否可执行。

### 3.2 `TaskEvent` 只追加历史

每条事件保存稳定 ID、单调序号、`commandId`、任务 ID、标题快照、事件类型、UTC 毫秒时间、原因、元数据和被撤销事件 ID。

本轮产生七类事件：

- `created`
- `title_updated`（metadata 保存 `beforeTitle` / `afterTitle`）
- `deadline_updated`（metadata 保存可空 `beforeDeadlineOn` / `afterDeadlineOn`）
- `completed`
- `completion_undone`
- `queue_reordered`
- `abandoned`（当前只用于删除，`metadata.action=delete`）

标题修改不会覆盖旧事件：`titleSnapshot` 与 `afterTitle` 保存新标题，`beforeTitle` 必须与此前历史链一致，当前任务投影最终也必须等于历史最后标题。期限修改同样使用后续事件纠正：从无期限设置、改期和清除都保留可空前后值，任务投影必须等于历史最后期限。撤销不会修改或删除原完成事件，而是追加一条指向原事件的新记录。重排事件保存移动任务、调整前完整顺序和调整后完整顺序，用于历史队首与最终队列回放。删除同样不物理删除任务或历史，而是追加 `abandoned` 事件并保留固定原因；完成、撤销和删除都保留任务已有期限字段。

### 3.3 `RewardTransaction` 奖励账本

- 完成事件关联一条 `+1`。
- 撤销事件关联一条 `-1`。
- `taskEventId` 唯一，避免同一事件重复发币。
- 当前余额始终由 `SUM(amount)` 得出；`balanceAfter` 只用于逐笔审计，必须与前缀和相同。
- 每个任务的奖励净值只能是 `0` 或 `1`。

本轮不实现喂食消费。未来加入消费后，如果撤销完成会使余额低于 `0`，撤销必须整笔失败且零写入，禁止用 `Math.max(0)` 静默截断。

### 3.4 `CommandReceipt` 幂等回执

每个写命令都携带稳定 `operationId`。数据库保存命令类型、请求指纹和原始结果：

- 相同 ID + 相同请求：返回原回执，`replayed=true`，不追加事件或金币。
- 相同 ID + 不同请求：返回 `COMMAND_ID_CONFLICT`，零写入。
- 不同 ID 再次完成已经完成的任务：返回状态冲突，零写入。

桌面前端把待确认命令保存到本地操作箱。应用重启、响应丢失或“命令已成功但快照刷新失败”后，都会先复用同一个 `operationId` 恢复原命令；恢复完成前统一锁住其他账本变更，较早的快照响应不能覆盖较新的界面。只有 `VALIDATION_ERROR`、`INVALID_TASK_STATE` 等明确领域拒绝才能判定命令未写入并清理操作箱；`STORAGE_ERROR`、完整性错误、版本错误和未知错误必须保留同一个 `operationId` 等待恢复确认。结构损坏的持久化操作不得自动删除；前端必须保留原记录并显式阻断新的账本写入。账本会话只有在内部运行锁释放后才发布 `READY`，使按钮启用、`canMutate()` 与前端就绪上报共享同一个可交互不变式。正常模式只有在 Rust 返回 `normal` 运行配置后才读写持久化操作箱；`--smoke` 使用内存中的临时操作，不读取、写入或清理正常模式的恢复凭据。

## 4. 六轮具体流程

### 第 1 轮：快速记录

输入：`operationId + title`。

```text
校验输入
→ 计算队尾位置
→ 新建 Task(pending)
→ 追加 created 事件
→ 保存命令回执
→ 同一事务提交
```

输出：任务 ID、事件 ID、当前任务 ID、金币余额。标题是唯一业务必填项；空标题和不可信 ID 会在进入数据库前被拒绝。

### 第 1.5 轮：修改未完成标题

输入：`operationId + taskId + title`。

```text
先检查是否已有回执
→ 确认 taskId 仍是立即 pending
→ 校验并更新 Task 标题与版本，不改变 queue_position
→ 追加 title_updated（beforeTitle / afterTitle）
→ 不创建奖励交易
→ 保存回执
→ 同一事务提交
```

客户端通过 `update_task_title({ operationId, taskId, title })` 提交有效变化；前端的空白或同名输入不提交。完整性检查按事件序列追踪当前标题，并对照事件 metadata、`titleSnapshot`、任务投影和命令回执。重复提交同一请求只重放回执，不能产生第二条标题事件。

### 第 1.75 轮：设置、修改或清除待办期限

输入：`operationId + taskId + deadlineOn`，其中 `deadlineOn` 只能是 `null` 或 `YYYY-MM-DD`。

```text
先检查是否已有回执
→ 确认 taskId 仍是当前可见的立即待办（status=pending、defer_until_ms=None、queue_position=Some）
→ 校验日历日并更新 Task.deadlineOn 与版本，不改变 queue_position
→ 追加 deadline_updated（beforeDeadlineOn / afterDeadlineOn）
→ 不创建奖励交易
→ 保存回执
→ 同一事务提交
```

客户端通过 `update_task_deadline({ operationId, taskId, deadlineOn })` 提交设置、改期或清除；相同值显式拒绝。期限独立于 `deferUntil`，不自动排序、隐藏、提醒、通知、同步日历或奖惩。今天、明天、`M/D`、跨年 `YYYY/M/D`、“逾期 N 天”和午夜刷新都由前端从持久化日历日派生，不写领域事件。

### 第 2 轮：完成任意待办

输入：`operationId + 用户勾选行的 taskId`。

```text
先检查是否已有回执
→ 确认 taskId 仍是立即可执行待办
→ Task 变为 completed
→ 追加 completed 事件
→ 追加 +1 奖励交易
→ 保存回执
→ 同一事务提交
```

客户端必须提交用户实际勾选的任务 ID，不能只发“完成当前任务”。完成非首项只移出目标任务，第一项与其他顺序保持不变。

### 第 3 轮：调整完整队列顺序

输入：`operationId + movedTaskId + expectedTaskIds + orderedTaskIds`。

```text
确认调整前顺序、任务集合、位置和版本
→ 所有队列位置暂移到唯一索引安全区
→ 按 orderedTaskIds 连续写回位置
→ 追加 queue_reordered 事件
→ 保存回执
→ 同一事务提交
```

`expectedTaskIds` 防止旧界面覆盖新顺序；结果未知时操作箱保留同一完整 payload 和 `operationId`。重排后第一项就是胶囊任务。

### 第 4 轮：删除待办

输入：`operationId + 用户删除行的 taskId`。

```text
先检查是否已有回执
→ 确认 taskId 仍是立即可执行待办
→ 通过 TaskWrite::Update 把 Task 变为 abandoned 并移出队列
→ 追加 abandoned 事件（reason=用户删除，metadata.action=delete）
→ 不创建奖励交易
→ 保存回执
→ 同一事务提交
```

客户端调用 `delete_task({ operationId, taskId })`，但账本不执行 SQL 物理删除；这样删除后仍可解释任务为何离开队列，也不会把删除误计入完成记录或金币。

### 第 5 轮：撤销完成

输入：`operationId + completionEventId`。

```text
读取原完成事件
→ 确认它仍是任务的有效完成
→ 确认余额扣回后不为负
→ Task 回 pending 队尾
→ 追加 completion_undone 事件
→ 追加 -1 奖励交易
→ 保存回执
→ 同一事务提交
```

输出仍包含当前任务 ID 和余额。原完成事件与 `+1` 交易继续保留，周报事实查询通过撤销关联排除这次完成。

## 5. 事务、不变式与异常恢复

SQLite 使用：

- `foreign_keys=ON`
- `journal_mode=WAL`（文件数据库）
- `synchronous=FULL`
- `busy_timeout=5s`
- `BEGIN IMMEDIATE`
- `PRAGMA application_id` 标识代办账本；内部数值保持不变，用于兼容已有数据
- `PRAGMA user_version=4` 标识当前源码结构版本

一次领域变化只能调用 SQLite 适配器的单个 `commit_transition`。适配器在同一个 `BEGIN IMMEDIATE` 事务内重新确认目标任务或完整队列、计算队尾和奖励前缀余额，再按顺序写任务、事件、奖励和回执；标题、期限、重排与删除也都只产生一个事务和一条事件，期限和删除的奖励变化固定为 `None`。任一步失败都会让事务回滚，没有分步公开写接口，也没有失败后退回内存数组或旧逻辑的静默降级。

复合快照、周报事实和完整性检查各自在单个只读事务中完成，不会拼接另一个进程提交前后的混合状态。v1、v2、v3 文件迁移前使用 `VACUUM INTO` 生成并验证 before-v4 一致性备份，再按受检版本路径升级到 v4。迁移后执行 SQLite `quick_check`、`foreign_key_check` 和领域一致性检查；完整性校验会按事件顺序回放入队、移出、重排、标题和期限历史，并与 `queue_position`、`deadlineOn` 投影及历史回执语义对照。任何不一致都会显式拒绝打开，原文件不会被空库覆盖。

```powershell
npm.cmd run desktop:check
```

`desktop:check` 会通过 npm 前置脚本先执行前端测试，然后运行 Rust 测试。发布配置与静态检查：

```powershell
cargo test --manifest-path .\src-tauri\Cargo.toml --release
cargo clippy --manifest-path .\src-tauri\Cargo.toml --all-targets -- -D warnings
```

异常退出账本冒烟，只使用系统临时目录：

```powershell
npm.cmd run ledger:smoke
```

桌面联合冒烟会短暂显示窗口，账本使用内存数据库，恢复操作箱也与正常模式隔离，不触碰正常应用数据：

```powershell
npm.cmd run desktop:smoke
```

开发运行会在 Tauri 应用数据目录创建 `zuoban-ledger.sqlite3`：

```powershell
npm.cmd run desktop:dev
```
