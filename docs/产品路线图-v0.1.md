> 状态：待办主流程已进入可试用状态，Windows x64 公开最新版为 `v0.1.2`；一级子代办、schema v5 与启动位置修复已经发布，`v0.1.1 → v0.1.2` 应用内升级和真实数据迁移仍待实机验证；宠物、AI 和日历暂停

## 1. 路线图总览

| 阶段 | 核心结果 | 主要验证问题 |
|---|---|---|
| 阶段 0 | 设计方案、极简原生主方案、状态矩阵与技术验证计划 | 产品闭环和第一版界面是否定义清楚 |
| 阶段 1 | 可日常使用的本地待办 MVP | 是否足够顺手、可靠，并能取代有道 |
| 阶段 2 | 非 AI 周回顾与日历只读时间线 | 时间背景是否帮助日常执行与回顾 |
| 阶段 3 | 可追溯的 AI 周报助手 | AI 是否真正减少周报整理成本 |
| 阶段 4 | 经真实需求选择的增强能力 | 哪些高级能力值得继续投入 |

核心闭环：

```text
快速记录
→ 较大事项可按需拆成一级子代办
→ 必要时在原行纠正未完成标题
→ 有明确日期时按需设置期限，无期限保持安静
→ 按顺序执行
→ 完成 / 删除录错项 / 延后 / 阻塞 / 放弃
→ 保存过程历史
→ 周末回顾
```

金币与宠物成长是后续可选奖励闭环，不再作为待办 MVP 的必经步骤或发布阻断项。

当前行内删除只负责清理录错或无需留在列表中的任务：内部保留软删除历史，但不计入完成记录。带原因的“真正不再做”、重新打开、回收站与永久清理仍属于后续状态闭环。

跨阶段不变原则：

1. 本地任务库是唯一事实来源。
2. 新应用彻底取代有道，不做长期同步或双数据源。
3. 记录时标题是唯一必填项。
4. 默认按用户队列顺序执行，不擅自重排。
5. 延后、阻塞和放弃不等于失败。
6. 宠物能力恢复后，只奖励、不惩罚，不死亡、不退化。
7. 任务规则、金币和成长由确定程序维护。
8. AI、日历或网络不可用时，核心应用仍完整可用。

## 3. 阶段 1：核心闭环 MVP

### 目标

交付一个可以替代有道、真正用于日常记录与执行的本地版本，先验证记录、顺序执行、状态分流和回顾是否足够顺手可靠。

### 迭代 1.1：可靠的本地待办

当前已实现：

- 快速新增，标题唯一必填，新任务追加到统一待办列表末尾。
- 待办页与完成记录页支持 `Ctrl+F` / 更多菜单搜索当前页面标题；共享搜索栏不占常态空间，命中片段高亮，`Esc` /“取消”退出。
- 未完成待办可双击标题或聚焦后按 `Enter` / `F2` 行内修改；`Enter` 或失焦保存，`Esc` 取消且不收起窗口。
- 原位编辑激活后才显示期限日期控件；默认 `deadlineOn=null`，清空恢复无期限。
- 无期限任务在常态列表和胶囊中不显示日期；已有期限按今天、明天、`M/D`、跨年 `YYYY/M/D` 或“逾期 N 天”紧凑显示。任务行标签可点击编辑；胶囊的独立 `edit-current-deadline` 按钮一次完成展开并聚焦日期输入，成功选择或清除后焦点回到标题。
- 每一行都有勾选入口，可以直接完成任意待办，不再设置独立的“完成当前任务”按钮。
- 支持鼠标拖动调整顺序，也支持聚焦某一行后按 `Alt+↑` / `Alt+↓` 上下移动。
- 列表第一项就是桌面胶囊显示的下一件事，不维护第二套“当前任务”选择。
- 每一行都有 `×`，可以一键软删除；内部以 `abandoned + metadata.action=delete` 保留历史，不发金币，也不进入完成记录。
- 标题修改追加 `title_updated` 事件并保留修改前后标题，不改变状态、顺序或金币；空白和未变化标题不提交。
- 期限修改通过 `update_task_deadline` 追加 `deadline_updated`，保留可空的修改前后期限；仅当前可见的立即待办可改，同值拒绝，完成、撤销和删除保留字段。
- 搜索只过滤账本快照的只读投影；搜索态仍可完成、修改、删除与撤销，但界面和控制器共同禁止局部重排。
- 标题、期限与顺序调整写入本地账本，应用重启后保持不变。
- 本地数据结构升级到新版本前，先生成迁移前备份。
- 期限独立于后续 `deferUntil`，不自动排序、隐藏、提醒、通知、同步日历或产生奖惩；逾期和午夜刷新只改变前端展示。

后续范围：

- 延后、阻塞、恢复与带原因的“真正不再做”。
- 重新打开、回收站恢复、永久清理及更完整的状态纠正。
- 完整任务事件历史与完整周报所需的更多查询；固定本地自然周的有效完成事实查询已经复用。
- 通用自动备份、用户主动导出和安全恢复；迁移前备份不等于这些通用能力已经完成。

退出条件：

- 应用异常退出后任务与顺序不丢失。
- 任意行完成和顺序调整均落到同一个本地事实来源。
- 标题修改只接受立即待办，修改后任务 ID、状态、顺序和金币不变，前后标题仍可追溯。
- 行内删除不会产生完成事实或奖励，但删除事件仍保留在本地账本中。
- 每项后续状态变化均可在历史中找到。
- 阻塞任务不会卡住后续任务。
- 多次重复操作不会产生重复事件或错误状态。

### 当前稳定切片：一级子代办（`v0.1.2` 已发布，`v0.1.3` 交互收口中）

当前本地工作区已经实现并通过统一门禁、账本 smoke 与桌面联合 smoke：

- 父行右侧固定低存在感 `＋` 作为唯一新增入口，有子项时以紧凑进度展开或收起；顶部快速记录仍只创建顶层代办。
- 单次添加态：非空标题按 `Enter` 或失焦创建，成功后退出并保持展开；空白失焦或 `Esc` 直接取消。零子项父项取消后收起空列表，已有子项时继续展开；子标题行内修改仍按 `Enter` 或失焦保存。
- 未完成子项软删除、完成与撤销、同组拖动和 `Alt+↑/↓` 重排。
- 搜索保留父上下文和真实进度；完成记录按父项分组并默认折叠；胶囊显示当前父项或“父项 / 当前子项”。
- 周回顾在父项本周完成时可带入更早完成且仍有效的子项；普通子完成仍必须发生在查询区间内。

父项勾选已实现为一次后端命令内自动完成剩余子项，再完成父项：子项完成事件零奖励，父项主事件最后写入并只产生一枚金币，整组写入同事务成功或回滚。第一版只支持一级，子项没有独立期限或金币，不能跨父项排序、改变父子关系、批量粘贴或由 AI 自动拆解。基础能力已进入 `v0.1.2`，新增入口与取消折叠的精简交互正在随 `v0.1.3` 完成发布验证。

### 迭代 1.2：桌面悬浮体验

范围：

- 按极简原生主方案实现和验收，不同时开发三套生产主题。

- 下一件任务胶囊态：始终取统一待办列表第一项；若它有未完成子项，则显示“父项 / 当前子项”和整组进度，圆圈完成屏幕上显示的动作。
- 展开任务面板：快速记录、一级父子组、可搜索和修改的标题、逐项勾选与彼此隔离的顶层/同组重排，以及可搜索的分组完成记录入口。
- 低频操作统一进入更多菜单或系统托盘。
- 从系统托盘打开时，始终进入统一展开的完整面板，不出现半展开窗口。
- 始终置顶、拖动与位置恢复。
- 全局快捷键快速记录。
- 临时隐藏到系统托盘；不再提供鼠标穿透。
- 全屏应用时按设置隐藏。
- 开机启动开关。

退出条件：

- 快速记录不会抢走不必要的焦点。
- 收起态不会长期遮挡工作内容。
- 从托盘打开后，完整面板尺寸与统一列表交互均正确恢复。
- 双击、`Enter` / `F2`、失焦、`Esc` 与输入法组合态下的标题编辑行为稳定，不误触窗口收起或重复提交。
- 勾选任意待办后列表保持连续；第一项变化时，胶囊同步显示新的下一件。
- 多显示器、缩放和重启后悬浮位置可恢复。

发布与更新状态：

- Windows x64 NSIS 安装器、Tauri 更新签名、版本一致性检查和 GitHub Actions 发布工作流已经实现，本机已成功生成 `setup.exe` 与 `.sig`。
- 正常模式启动后检查更新，并每 24 小时再次检查；更多菜单可手动检查。用户明确点击安装后才下载、安装并重启。
- smoke 前端不启动更新控制器，Rust 也会在访问网络前拒绝更新请求。
- `v0.1.2` GitHub Release 已发布，公开安装器、线上 `latest.json`、更新签名和资产摘要由发布工作流闭环验证；当前用户安装基线仍为 `0.1.1`，完整自动升级与 schema v4→v5 迁移仍待实机验证。Windows Authenticode 暂未配置，可能出现 SmartScreen 提示。

### 暂停里程碑：金币与宠物成长

这一里程碑保留设计，不阻断当前待办 MVP。只有待办核心完成真实试用后，才重新决定是否进入实现。

范围：

- 每项任务首次有效完成奖励 1 枚金币。
- 奖励撤销与独立金币账本。
- 两种以内的基础食物。
- 喂食、成长值与三个成长阶段。
- 待机、完成、喂食和成长反馈。
- 今日完成数、金币余额和成长进度。

退出条件：

- 同一任务不会因重复点击、重启或事件重放而重复发币。
- 喂食后的金币、成长值和账本完全一致。
- 宠物暂停数日后不会死亡、退化或丢失成长。
- 动画不会阻塞任务操作。

### 迭代 1.4：历史与非 AI 周报

当前已实现独立完成记录页。`v0.1.2` 的一级子代办把完成记录改为父项分组且默认折叠，并扩展了本周复制：父项在本周完成时只输出一次，可把更早完成且仍有效的子项作为缩进明细；父项未完成时，普通子项仍须在本周完成才单独输出。

当前已实现：

- 固定查询 `[本周一 00:00, 下周一 00:00)`，复用现有 `weekly_facts`，不新增任务状态或数据库结构。
- 只输出未被撤销的有效完成事件，格式为 `## 本周完成` 和编号列表。
- 本周为空时明确提示且不覆盖剪贴板；非空时提示复制项数。
- 全程读取本地账本，不依赖 AI 或周报网络服务。

后续范围：

- 今天、上周和自定义时间查询。
- 完成、进行中、阻塞和计划调整分组。
- 单项任务完整过程时间线。
- 可选择纳入项。
- 复制纯文本或包含更多分组的 Markdown 周报。
- 本周完成数、金币收入和成长变化。

退出条件：

- 完成后又重开的任务不会被错误算作当前有效完成。
- 阻塞、放弃和延后原因正确进入回顾。
- 周跨月、跨年时的统计范围正确。
- 周报生成完全不依赖 AI 和网络。

### 阶段 1 明确不做

- 不接入有道，也不提供有道同步或历史迁移能力。
- 不做账号与云端任务库。
- 不做番茄钟、AI 和日历；本地可选期限不代表日历能力恢复。
- 不做复杂优先级算法和自动排序。
- 不做多级子代办、跨父项移动、父子转换、子项期限或子项金币。
- 不做批量子项粘贴、多选操作或 AI 自动拆解。
- 不做连续签到和宠物惩罚。
- 不做团队协作和复杂项目管理。
- 不做大量宠物、家具、服装、抽取和多货币。
- 不并行交付极简原生、深色透明和像素桌宠三套完整生产主题。

### 阶段 1 发布判定

只有同时满足以下条件，才能称为第一版完成：

1. 快速记录足够顺手。
2. 顺序执行不会被阻塞任务卡住。
3. 做不了的任务有明确去处。
4. 历史能够真实还原本周发生的事情。
5. 应用重启、断网或异常退出不会破坏数据。
6. 不依赖有道、宠物、AI、日历和其他外部服务即可完成整个待办闭环。
7. 完成至少一个完整工作周的真实自用试运行。
8. 首个 GitHub Release 可完成 Windows x64 手动首次安装，并通过一个后续版本实机验证完整自动更新闭环。

## 4. 暂停候选：回顾增强与日历轻量联动

本阶段尚未排期，只有待办 MVP 稳定使用后才重新评估。

### 进入条件

- 阶段 1 稳定使用至少一个完整工作周。
- 核心任务与奖励数据没有阻断使用的可靠性问题。
- 用户已明确选择第一种需要读取的日历来源。

### 目标

在不改变待办事实的前提下，为当天执行和每周回顾补充固定日程背景。

### 范围

- 只读显示今天与近期日程。
- 悬浮窗显示下一项日程。
- “待办 + 日程”今日时间线。
- 用户主动选择允许读取的日历。
- 手动将日程复制成一条独立待办。
- 更完整的非 AI 周回顾筛选与模板。

### 明确不做

- 不创建、修改或删除外部日历事件。
- 不做日历双向同步。
- 不自动把所有日程转成待办。
- 不因会议结束自动完成任务或发金币。
- 不同时接入多个日历平台。
- 不加入 AI 周报。

### 退出条件

- 日历授权、读取、刷新和断开流程可用。
- 全天、跨日、重复事件和时区显示正确。
- 日历不可用时，待办、历史与宠物照常运行。
- 今日时间线能够清楚区分任务和日程。

## 5. 暂停候选：AI 周报与回顾助手

本阶段尚未排期，不影响核心待办离线运行。

### 进入条件

- 非 AI 周报已经稳定并积累真实使用样本。
- 已明确 AI 服务、费用、隐私与数据保留策略。
- 已定义事实准确性和来源追溯的验收方式。

### 目标

利用真实任务历史生成可检查、可修改、可追溯的周报草稿，减少回忆和整理成本。

### 范围

- 用户选择本次允许使用的数据范围。
- AI 生成“本周完成、进行中、阻塞、计划调整、下周建议”。
- 每条事实可跳回来源任务或日程。
- 草稿单独保存，不覆盖历史。
- AI 周报稳定后，小步加入任务拆分与标题整理建议。

### 明确不做

- AI 不自动完成、删除、延后、放弃任务。
- AI 不改变任务顺序。
- AI 不决定金币和成长值。
- AI 不虚构完成事项。
- 不做无限制宠物自由聊天和全天候主动代理。
- 不让 AI 结果成为唯一历史依据。

### 退出条件

- 一个完整工作周的周报事实可以追溯。
- 用户可以排除某项数据并重新生成。
- AI 不可用时仍可使用基础周回顾。
- 真实试用确认 AI 确实降低了周报整理工作量。

## 6. 阶段 4：经验证后的可选增强

候选方向：

- 日历事件创建与带冲突策略的双向联动。
- 多日历来源。
- 自然语言识别日期、项目和标签。
- AI 任务拆分与顺序建议。
- 更多宠物、食物、玩具、房间和装扮。
- 宠物根据真实历史进行有限度交流。
- 多设备同步与备份。
- 月度、季度回顾和更多导出格式。
- 深色透明与像素桌宠候选皮肤适配。

每项能力必须单独立项、独立验收。不能把多个高风险能力一次性合并上线，也不能为了增加功能数量破坏悬浮窗的简洁性。
