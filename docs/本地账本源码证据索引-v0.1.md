# 代办：本地账本源码证据索引 v0.1

> 更新时间：2026-07-23
> 用途：保留影响账本架构、事务、恢复和迁移策略的官方证据，后续会话优先读取本页，不重复追完整源码。

## 1. `rusqlite` 选择

| 官方资料 | 核心行为 | 本项目落点 |
|---|---|---|
| [rusqlite 仓库与 README](https://github.com/rusqlite/rusqlite) | Rust 的 SQLite 轻量封装；`bundled` 可编译随包 SQLite | `Cargo.toml` 使用 `rusqlite 0.40.1` 和 `bundled` |
| [rusqlite 0.40.1 功能开关](https://docs.rs/crate/rusqlite/0.40.1/features) | `bundled` 通过 `libsqlite3-sys` 构建内置 SQLite | Windows 不依赖外部 `sqlite3.dll` |
| [Connection 官方 API](https://docs.rs/rusqlite/0.40.1/rusqlite/struct.Connection.html) | 单连接执行 SQL、参数绑定、事务和 PRAGMA；连接可移动但不应无保护共享 | `SqliteLedgerStore` 持有一个连接，Tauri 状态外层使用 `Mutex` |
| [Transaction 官方 API](https://docs.rs/rusqlite/0.40.1/rusqlite/struct.Transaction.html) | 显式 `commit` 提交；未提交事务离开作用域时回滚 | 故障注入返回错误后依赖事务回滚，不补写旧链路 |

结论：每个应用实例持有一个 SQLite 连接，预期负载是单机、低并发和短事务；适配层仍使用事务处理多个实例的跨连接竞争。当前没有异步连接池需求。SQLx 的异步、宏和迁移设施更完整，但会增加运行时、编译和学习成本；等到出现持续高并发或复杂迁移需求时再重新评估。

## 2. SQLite 事务与恢复

| 官方资料 | 核心行为 | 本项目落点 |
|---|---|---|
| [SQLite 事务文档](https://www.sqlite.org/lang_transaction.html) | 显式事务将一组修改作为提交或回滚单元；`IMMEDIATE` 会立即开始写事务 | `commit_transition` 使用 `BEGIN IMMEDIATE`；标题/期限修改、父项与伴随子项完成或重排完整队列时，在事务内复查回执、目标状态、活动子项集合、投影和任务版本，再一并提交投影、全部事件、奖励与主回执 |
| [SQLite WAL 文档](https://www.sqlite.org/wal.html) | WAL 将变更先追加到日志，崩溃恢复和读写并行语义与回滚日志不同 | 文件数据库启用 `journal_mode=WAL`，异常退出后由新连接恢复 |
| [SQLite 原子提交说明](https://www.sqlite.org/atomiccommit.html) | 事务和日志协议用于保证掉电或进程异常下的原子提交 | 子进程分别在提交前和提交后、响应返回前退出；父进程重开验证完整回滚或同 ID 回执重放 |
| [SQLite PRAGMA quick_check](https://www.sqlite.org/pragma.html#pragma_quick_check) | 快速检查表与索引的低层一致性 | 身份与结构版本检查通过、迁移结束后执行 `quick_check(1)` |
| [SQLite PRAGMA foreign_key_check](https://www.sqlite.org/pragma.html#pragma_foreign_key_check) | 返回违反外键约束的记录 | 打开受支持账本时检查孤立任务事件和奖励关联 |
| [SQLite 外键 PRAGMA](https://www.sqlite.org/pragma.html#pragma_foreign_keys) | `foreign_keys` 只能在事务外切换，事务内修改不会生效 | v1→v2、v2→v3、v3→v4、v4→v5 的受控迁移在事务外切换外键设置，迁移提交后恢复，并用 `foreign_key_check` 验证 |
| [SQLite `VACUUM INTO`](https://www.sqlite.org/lang_vacuum.html#vacuum_with_an_into_clause) | 可生成与原库一致、体积紧凑的独立数据库文件 | v1、v2、v3 或 v4 磁盘账本迁移到 v5 前生成并验证 before-v5 一致性备份，回读 `quick_check` 与原版本后才开始迁移 |

项目不会只依赖 SQLite 低层检查；其后还会完整反序列化每条幂等回执，并按事件序列回放任务队列、标题历史和期限历史，验证命令主键、任务、事件、奖励交易、重排前后集合、事件时点奖励前缀余额和历史第一项，同时检查每个任务有且只有一个创建历史起点、每任务奖励净值、任务状态投影、最终队列位置、最终可空期限和有效完成事件。复合快照、周报事实和完整性报告分别使用单个只读事务，避免拼接跨提交状态。

## 3. 数据库身份与版本

| 官方资料 | 核心行为 | 本项目落点 |
|---|---|---|
| [SQLite `application_id`](https://www.sqlite.org/pragma.html#pragma_application_id) | 应用可以在数据库头中写入自己的文件格式标识 | 使用 `0x5A424E31`；非零且不匹配时拒绝打开 |
| [SQLite `user_version`](https://www.sqlite.org/pragma.html#pragma_user_version) | 应用可自行使用数据库头中的整数版本字段 | 当前源码结构版本为 `5`；v1、v2、v3、v4 按级迁移，高于当前支持版本时快速失败 |
| [SQLite ALTER TABLE](https://www.sqlite.org/lang_altertable.html#making_other_kinds_of_table_schema_changes) | 对 CHECK 等非直接 ALTER 的结构变化，应新建表、复制数据、删除旧表并重命名，再做一致性检查 | v1→v2 增加 `queue_reordered`、v2→v3 增加 `title_updated`、v3→v4 增加可空期限、v4→v5 增加父子字段与子代办事件；迁移核对旧数据、序号、索引、外键和记录 |

全新数据库直接建立 v5 结构；v1、v2、v3、v4 磁盘文件先生成并验证 before-v5 一致性备份，再用内嵌、有序 SQL 和 `BEGIN IMMEDIATE` 按级迁移到 v5。身份、版本和用户对象检查都在迁移事务内完成；两个独立连接首次打开同一路径时会串行初始化。只有完全空白、无应用标识的数据库才允许初始化，已有未知用户对象的 SQLite 文件会显式拒绝接管。迁移后的 v5 数据不支持交给 v4 客户端降级读取。当结构继续升级、需要校验和、多条跨版本路径和可重复迁移时，应替换为成熟迁移器；替换仍必须留在 SQLite 适配层，不能污染领域模型。

## 4. 工程边界与测试入口

| 资料或代码证据 | 已确认行为 | 本项目落点 |
|---|---|---|
| [Tauri 前端配置](https://v2.tauri.app/start/frontend/) | Tauri 可承载静态前端资源，不要求 React、Vue 或额外打包器 | `desktop/app.js` 使用原生 ES modules 组合 `state`、`selectors`、`views`、`ledger-session`、窗口/宠物组件和 `infrastructure` 适配器 |
| [Tauri `dragDropEnabled` 配置](https://v2.tauri.app/reference/config/#dragdropenabled) | Windows 上要使用网页原生 HTML5 拖放，应关闭 Tauri 自己的文件拖放接管 | 窗口配置保持 `dragDropEnabled: false`；任务排序柄使用 HTML5 拖放，同时提供 `Alt + ↑/↓` 键盘路径 |
| [Node.js 测试运行器](https://nodejs.org/api/test.html) | `node --test` 可直接执行匹配到的测试文件 | `npm.cmd run frontend:check` 执行 `desktop/tests/*.test.js`，没有新增测试框架依赖 |
| `src-tauri/src/app.rs` | 应用组合根集中管理状态、插件、命令注册和启动接线 | 账本领域不依赖 Tauri；桌面、运行配置、前端探针和账本命令都用模块限定路径注册 |
| `src-tauri/src/frontend_probe.rs` + `integration_smoke.rs` | Rust 会等待真实前端报告，不再用“窗口已打开”代替“账本界面已就绪” | 前端完成运行配置确认、操作箱恢复和快照读取，并在内部运行锁释放后发布 `READY`，再调用 `report_frontend_ready`；冒烟校验 `profile=smoke`、`ledgerReady=true` |
| `desktop/app/infrastructure/outbox-store.js` + `desktop/app/ledger-session.js` | 结构损坏的持久化操作不得被当作“没有待确认操作” | 保留原始记录、显式阻断新写入；只能由明确领域拒绝或确认完成的恢复流程清理操作箱 |
| `src-tauri/src/ledger/sqlite.rs` + `sqlite/schema.rs` + `sqlite/integrity.rs` | 原子提交、结构迁移和完整性校验是同一 SQLite 适配器的不同职责 | 领域与应用服务继续只依赖项目自己的 `LedgerStore` 接口 |
| `TaskListController` → `LedgerSession.updateTaskTitle` → `update_task_title` → `TaskService::update_task_title` → `update_task_title_transition` | 行内编辑只提交意图；可靠写链路用稳定 `operationId` 保存标题投影、事件和回执 | `title_updated` 保存 `beforeTitle` / `afterTitle`，仅立即待办可改；不改变状态、顺序或金币，完整性检查回放标题历史链 |
| `TaskListController` → `LedgerSession.updateTaskDeadline` → `update_task_deadline` → `TaskService::update_task_deadline` → 期限领域转换 | 日期控件和期限标签只提交意图；可靠写链路用稳定 `operationId` 保存可空期限投影、事件和回执 | `deadline_updated` 保存 `beforeDeadlineOn` / `afterDeadlineOn`，仅当前可见的立即待办可改、同值拒绝；不改变状态、顺序或金币，完整性检查回放期限历史链 |
| `desktop/app/ledger-session.js` → `ledger/commands.rs` → `TaskService::delete_task` → `delete_task_transition` | 删除使用与其他写命令相同的稳定 `operationId`、操作箱和单事务链路 | `delete_task({ operationId, taskId })` 通过 `TaskWrite::Update` 写成 `abandoned`，固定保留“用户删除”原因和 `metadata.action=delete`，不物理删除、不产生奖励 |
| `src-tauri/src/desktop/window.rs` + `tray.rs` + 前端 `TauriGateway` | 窗口实际状态由 Rust 决定并通过事件同步；托盘恢复入口使用同一窗口动作 | 模式变化发布 `window-status-changed`，前端据此更新 DOM；托盘菜单和托盘单击都调用 `restore_expanded` |
| `src-tauri/src/ledger/smoke.rs` + `smoke/tests.rs` | 运行时故障注入/进程强退与普通自动测试已经分离 | CLI 冒烟验证真实异常退出；测试模块验证领域、迁移、并发和损坏数据边界 |

Tauri 命令注册的已验证经验：命令定义在独立模块时，`tauri::generate_handler!` 中保留 `crate::模块::命令函数` 的完整限定路径。命令宏生成的配套符号仍在定义模块中；直接把命令函数导入组合根再用短名称注册，会让 `generate_handler!` 在错误的模块路径下解析配套符号并导致编译失败。后续新增命令继续在所属模块定义，在 `app.rs` 用模块限定路径统一注册；不复制一套包装命令，也不设置旧入口静默回退。

## 5. 已验证结论与后续落点

- 当前反馈代码已完成 155 个前端与发布文档测试与 66 个 Rust 测试；独立账本和最终桌面联合冒烟均为 `passed=true`，桌面报告包含 `scaleFactor=1.25`、父项级联完成、奖励规则、回放稳定性和子项撤销守卫。
- `rusqlite + bundled` 已在 Windows 验证链路中编译通过；v1/v2/v3/v4 → v5 磁盘迁移已用合成填充样本验证迁移前一致性备份、迁移后 `quick_check`、外键、层级约束和级联完成。公开证据只保留稳定验证结论，不记录本地备份文件名或真实账本统计。
- 标题与期限修改、一级子代办、父项级联完成、历史保留型删除、两种重排、事务回滚、故障注入、响应丢失重放、跨连接写入、迁移后旧事件/回执继续可用、级联索引与历史投影篡改拒绝、WAL 重开、外键与回执一致性、拒绝接管未知库和未知版本拒绝均已有自动测试或独立冒烟证据。
- 当前源码已经具备 v1、v2、v3、v4 升级到 v5 前的一次性一致性备份；尚没有面向用户的通用自动备份、导出和恢复交换协议，不能把迁移保护表述成完整备份产品。
- 下一次设计通用备份与恢复时，应继续查询 SQLite 官方在线备份 API 与 `VACUUM INTO` 的最新约束，并另建证据条目。
