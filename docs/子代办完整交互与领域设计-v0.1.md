# 代办：子代办完整交互与领域设计 v0.1

> 状态：产品与实现契约已冻结；实现已通过统一门禁、账本 smoke 与桌面联合 smoke，并随 `0.1.2` 正式发布（2026-07-24）
>
> 原型入口：[`docs/prototypes/subtasks-v0.1/index.html`](./prototypes/subtasks-v0.1/index.html)
>
> 术语：面向用户统一称“子代办”；代码标识可使用 `subtask`

## 1. 结论先行

子代办可以加入，但第一版必须把复杂度收在展开态和领域层，不能重新挤满主界面。

冻结以下产品口径：

1. 只支持一级子代办，不允许子代办继续嵌套。
2. 顶层代办仍是主队列、整体期限、最终完成和金币奖励的主体。
3. 每个父待办行固定一个低存在感 `＋` 作为唯一新增入口；已有子代办时再显示紧凑进度，例如 `1/3`，由进度负责展开或收起。
4. 子代办可以添加、修改、软删除、完成、撤销完成和同组重排。
5. 子代办完成后保留在父项原位置，不自动移到底部，并写入可回顾的完成事件。
6. 子代办完成不发金币；父代办最终完成仍只发 `1` 枚金币。
7. 用户勾选父代办时，后端在同一事务内自动完成全部未完成有效子代办，再完成父代办；不弹确认框。
8. 完成最后一个子代办不会自动完成父代办，必须由用户再次勾选父代办确认整件事结束。
9. 删除含子代办的父项需要一次危险操作确认；确认后父项软删除，整个任务组从当前列表隐藏，子代办和已有完成历史不物理删除。
10. 桌面胶囊显示当前父代办中的第一条未完成子代办；如果子代办均完成，则重新显示父代办。
11. 完成记录中的父子组默认折叠，只显示父标题、真实进度、时间和展开入口；搜索命中子代办时临时展开对应组。
12. 第一版期限只属于顶层代办，子代办不单独设置期限、提醒或日历联动。
13. 第一版不支持跨父项移动、父子转换、批量粘贴、多选和自动拆解。

### 1.1 兼容协议冻结

- 数据库版本固定升级为 `v5`；旧 `v4` 数据全部迁移为顶层代办，迁移前继续执行现有备份与校验流程。
- 快照只做加法：新增 `subtasks`；`currentTask`、`queue` 与 `completed` 的既有语义不变，且仍只包含顶层代办。
- 任务序列化新增 `parentTaskId` 与 `siblingPosition`；顶层两者均为 `null`，子代办的 `queuePosition` 必须为 `null`。
- 新增稳定命令 `create_subtask` 与 `reorder_subtasks`，前端载荷分别固定为 `{ parentTaskId, title }` 和 `{ parentTaskId, movedTaskId, expectedTaskIds, orderedTaskIds }`；`operationId` 仍由现有可靠写入层统一保存与注入。
- 子代办事件父上下文固定写入 `metadata.parentTaskId` 与 `metadata.parentTitle`；重排事件额外写入 `movedTaskId`、`beforeTaskIds`、`afterTaskIds`。
- 继续复用现有稳定错误码 `VALIDATION_ERROR`、`INVALID_TASK_STATE`、`TASK_NOT_FOUND`、`TASK_EVENT_NOT_FOUND`、`COMMAND_ID_CONFLICT` 与 `CONCURRENT_MODIFICATION`，不为界面分支新增另一套错误协议。
- 旧命令、旧操作箱载荷和旧幂等回执必须继续有效；不得把 `capture_task` 改成带可选父 ID 的兼容分支，也不得保留静默旧链路回退。
- `complete_task` 的 IPC 和操作箱载荷不变；目标为父项时，唯一外部回执继续指向最后写入的父完成主事件，伴随子完成事件通过受检级联元数据关联。

## 2. 目标与边界

### 2.1 目标

- 把“写周报”这类较大的代办拆成明确动作，同时保持桌面只提示一件真正要做的事。
- 让添加、修改、删除和排序都能在当前窗口完成，不增加独立详情页。
- 让子代办完成事实进入历史，为周报和回顾提供依据。
- 保持现有可靠写入链路：界面只提交意图，Rust 领域层决定状态，SQLite 保存唯一事实。
- 保持快速记录足够快：顶部“记一件事”仍然只创建顶层代办，不强迫用户先决定层级。

### 2.2 第一版不做

- 二级及更深层级。
- 看板、项目、标签、负责人、备注和附件。
- 把顶层代办拖成子代办，或把子代办拖成顶层代办。
- 把子代办拖到另一个父代办。
- 子代办独立期限、提醒、日历同步和金币。
- 自动完成父代办。
- AI 自动拆解。
- 多行粘贴后自动批量创建。

## 3. 信息结构与常态展示

### 3.1 收起状态

没有子代办时不显示无意义的 `0/0`，但父行右侧保留固定新增入口：

```text
○ 回复项目反馈                           ＋  ×  ⠿
```

存在子代办时，在标题和期限之间增加进度入口，右侧新增入口位置不变：

```text
○ 写周报                    1/3  7/25  ＋  ×  ⠿
```

- `1/3` 是按钮，也是展开入口。
- `＋` 是唯一新增入口；正常态、父项修改态和子项新增态都保留它的网格位置，新增过程中只禁用，不让删除与排序按钮横向跳动。
- 期限仍然是内容区最后一个信息，不让进度把日期挤回标题附近。
- 删除和排序仍位于最右侧操作区，不与进度、期限混成一组。
- 搜索期间隐藏 `＋`，因为筛选投影中不允许新增子代办。

### 3.2 展开状态

```text
○ 写周报                    1/3  7/25  ＋  ×  ⠿
│  ● 汇总本周完成                         ×  ⠿
│  ○ 整理本周问题                         ×  ⠿
│  ○ 写下周计划                           ×  ⠿
```

- 子列表只做一级缩进和一条淡分组线，不做卡片套卡片。
- 完成子代办降低对比度并加删除线，但保留原顺序。
- 子列表没有独立滚动条；整个待办列表仍只使用现有内部滚动区。
- 展开状态是前端临时状态，不写领域事件；应用重启后默认全部收起。
- 当前队首父代办可以在展开窗口中保留最近的展开状态。

### 3.3 当前动作

顶层队列仍只有父代办，但界面派生一个“当前动作”：

```text
当前父代办 = 顶层队列第一项

若存在未完成子代办：
  当前动作 = 子代办列表中的第一条未完成项
否则：
  当前动作 = 父代办本身
```

这个派生规则只影响胶囊显示和展开后的焦点，不新增第二套队列事实。

## 4. 添加子代办

### 4.1 添加第一条

无子代办的父行右侧固定显示低存在感 `＋`。它与删除、排序入口各占独立操作列，不依赖悬停才出现。

点击后：

1. 父行下方展开子列表区域。
2. 出现单行输入框“子代办名称”。
3. 焦点直接进入输入框。
4. 非空标题按 `Enter` 或输入框失焦时创建。
5. 创建成功后退出输入态，第一条子代办留在展开列表中，父行固定 `＋` 恢复可用；创建失败保留草稿和输入态。
6. 空白或纯空格失焦时静默退出，不创建也不报错；`Esc` 同样取消添加。因为此时没有旧子项，取消后同时收起空的子列表区域。

父代办修改态继续使用同一枚固定 `＋`，不再在编辑区域内增加第二个文字入口。

### 4.2 再次添加

已有子代办时仍点击父行固定 `＋`，输入框插在子列表末尾。取消空白输入或按 `Esc` 只退出新增态，已有列表保持展开；创建成功后也保持展开。子列表底部不再重复显示“添加子代办”。

### 4.3 快速记录边界

顶部“记一件事”永远创建顶层代办。第一版不解析以下隐式语法：

- 前导空格表示子代办。
- `-`、`*` 或编号表示子代办。
- 粘贴多行自动创建多个子代办。

这样可以避免误判，也不改变当前三秒内完成记录的主流程。

### 4.4 写入失败

| 状态 | 界面行为 |
|---|---|
| 正在保存 | 保留输入文字，禁用该输入框和父行固定 `＋`，显示克制等待态 |
| 确定性拒绝 | 输入框保持打开，标出问题，不清空内容 |
| 结果未知 | 保留原 `operationId`，锁住后续写入，提示“正在确认刚才的操作” |
| 重试成功 | 用真实账本快照重绘并清空草稿，退出输入态并恢复父行固定 `＋` |

## 5. 修改

### 5.1 修改父代办

继续复用当前交互：

- 双击父标题，或聚焦后按 `Enter` / `F2`。
- 原位显示标题和可选期限。
- `Enter` 或失焦保存，`Esc` 取消。
- 修改态继续保留父行固定 `＋`，不在编辑区复制第二个新增入口，也不新增详情页。

### 5.2 修改子代办

- 双击子标题，或聚焦后按 `Enter` / `F2`。
- 只显示单行标题输入框。
- `Enter` 或失焦保存，`Esc` 取消。
- 空白标题不提交；未变化标题直接取消。
- 中文输入法组合期间，`Enter` 和 `Esc` 不得提前保存或退出。
- 已完成子代办不能直接修改；必须先撤销完成。

### 5.3 第一版不可修改的关系

- 不能把父代办改成子代办。
- 不能把子代办提升为顶层代办。
- 不能更换子代办所属父项。
- 不能为子代办添加子代办。

## 6. 完成与撤销

### 6.1 完成子代办

点击子代办圆圈：

```text
pending 子代办
→ 写入子代办完成事件
→ 不产生金币
→ 子代办留在原位置并显示完成态
→ 父项进度从 1/3 变为 2/3
→ 当前动作切换到下一条未完成子代办
```

完成子代办不是完成父代办，也不改变顶层队列顺序。

### 6.2 撤销子代办完成

只要父代办仍然处于待办状态，可以通过两处撤销：

- 再次点击展开列表中的已完成圆圈。
- 在完成记录中点击该子代办行末 `↶`。

撤销后：

- 追加撤销事件，不删除旧完成事件。
- 子代办回到未完成态和原顺序。
- 不发生金币回退，因为子代办完成从未发币。

父代办已经完成时，不允许直接撤销其子代办。必须先撤销父代办完成，避免出现“父项已完成、子项却未完成”的非法状态。

### 6.3 完成父代办

父代办没有子代办时，保持当前完成规则。

父代办存在子代办时：

- 点击父圆圈只提交一次父项 `complete_task` 意图，前端不循环勾选子项。
- 后端为全部仍为 `pending` 的有效子代办追加真实 `subtask_completed` 事件；已经完成的子项不重复写，已软删除子项忽略。
- 子项级联完成仍不产生金币；父项主完成事件最后写入，并只产生一枚金币。
- 最后一条子代办单独完成后不会自动完成父项；用户仍必须明确再勾选一次父项。

父代办完成时：

```text
父 pending
→ 全部 pending 有效子代办 completed（每项零奖励）
→ 父 completed
→ 全部子完成事件、父完成主事件、金币 +1 与唯一主回执同事务提交
→ 整组离开当前待办列表
→ 完成记录保留父项和全部子项完成事实
```

### 6.4 撤销父代办完成

- 继续在完成记录父行末使用 `↶`。
- 父代办回到顶层队列末尾。
- 子代办保持撤销前各自的完成状态和顺序。
- 父完成奖励同步 `-1`，净奖励保持正确。

## 7. 删除

### 7.1 删除未完成子代办

- 行末低存在感 `×` 只在悬停或键盘聚焦时完整显示。
- 点击后执行可追溯软删除，进度分母立即排除该子代办。
- 不产生金币，不进入“完成记录”。
- 已完成子代办不能直接删除；需要先撤销完成，再删除。
- 第一版不增加删除撤销入口，保持与现有顶层删除一致。

### 7.2 删除没有子代办的父项

保持当前规则：点击 `×` 直接软删除，不额外确认。

### 7.3 删除含子代办的父项

这是唯一新增确认的删除场景。

确认层文案：

```text
删除“写周报”？

整个任务组将从当前列表移除。
已有完成记录仍然保留。

[取消]  [删除整个任务组]
```

领域实现不逐条连续删除子代办：

1. 只把父代办写为 `abandoned`。
2. 子代办投影、顺序和事件全部保留。
3. 当前列表按父项状态隐藏整个组。
4. 完成历史仍可读取已经完成的子代办。
5. 后续如果增加“重新打开”，可以完整恢复原任务组。

这样仍然是一条任务写入、一个事件和一份幂等回执，不会发生父项已删但只删掉一半子项的中间状态。

第一版不提供“删除父项但把子代办提升为普通代办”，因为它会同时改变层级、顶层队列位置和当前动作，风险远高于单纯删除。

## 8. 排序

### 8.1 顶层排序

- 拖动父代办时，整个父子组作为一个单位移动。
- 顶层 `queuePosition` 仍只记录父代办。
- 键盘继续使用 `Alt + ↑/↓`。

### 8.2 子代办排序

- 子代办有自己的低存在感排序柄。
- 只能在当前父代办内部排序。
- 键盘同样使用 `Alt + ↑/↓`，但作用域限定为同一子列表。
- 完成子代办仍参与排序并保留位置。
- 第一版拒绝跨父项拖动和拖出成为顶层代办。

### 8.3 搜索期间

搜索是局部投影，因此父项和子项排序全部禁用，不能把筛选结果提交成完整顺序。

## 9. 搜索

搜索同时匹配父标题和子标题，但必须保留层级上下文。

| 命中情况 | 展示规则 |
|---|---|
| 只命中父标题 | 显示父行；保持进入搜索前的收起状态，用户可主动展开全部子代办 |
| 只命中子标题 | 显示父行作为上下文，并临时展开命中的子代办 |
| 父子都命中 | 显示父行和完整子列表 |
| 无命中 | 显示“没有找到相关待办” |

其他规则：

- 进度始终显示整组真实值，例如只显示一条命中项时仍显示 `2/5`，不能显示 `1/1`。
- 搜索临时展开不覆盖用户进入搜索前的展开状态。
- 搜索期间仍允许完成、撤销、修改和删除当前可见项。
- 搜索期间不允许添加子代办，避免创建后因不匹配而立刻消失。
- 搜索期间不允许任何层级的重排。
- 完成记录搜索同时匹配父标题快照和子标题快照。

## 10. 桌面胶囊

### 10.1 显示规则

父代办仍有待处理子代办时：

```text
○ 写周报 / 整理本周问题       1/3  7/25  ˅
```

- 标题区域使用一行父子路径，超长时统一省略。
- 圆圈完成屏幕上显示的子代办，不是父代办。
- `1/3` 显示整组真实进度。
- 期限仍使用父代办期限，并位于信息区最后。

所有子代办均完成后：

```text
○ 写周报                         3/3  7/25  ˅
```

此时圆圈完成父代办。

### 10.2 展开规则

点击胶囊标题：

1. 展开主窗口。
2. 定位并展开当前父代办。
3. 焦点落到胶囊刚才显示的子代办圆圈。
4. 不再无条件把焦点送到顶部快速记录输入框。

## 11. 完成记录与周报

### 11.1 完成记录

完成记录按父代办归组，并且默认折叠。进入页面时只显示一行摘要：

```text
写周报                    进行中 · 2/3  ›
```

父代办最终完成后，摘要同时保留整组进度、完成时间和父项撤销入口：

```text
写周报                  3/3 · 7/23 17:40  ›  ↶
```

点击 `›` 才展开子代办完成明细：

```text
写周报                  3/3 · 7/23 17:40  ⌄  ↶
│  汇总本周完成                 7/23 09:40
│  整理本周问题                 7/23 11:20
│  写下周计划                   7/23 17:30
```

- 每次进入完成记录页时，所有父子组默认折叠。
- 展开状态只是当前界面临时状态，不写入数据库。
- 独立代办仍保持普通单行，不增加无意义的展开箭头。
- 搜索只命中子代办时，对应父项临时自动展开并只显示命中明细；退出搜索后恢复搜索前的折叠状态。
- 父项未完成时，子代办可直接撤销。
- 父项完成后，子代办撤销入口禁用并提示“请先撤销父代办”。
- 父项被删除后，已有子代办完成记录仍保留，但不能从该处直接改动已隐藏任务组。

### 11.2 历史标题快照

子代办完成事件必须同时保存：

- 子代办标题快照。
- 父代办 ID。
- 父代办标题快照。

后续修改父标题或子标题不能篡改过去的完成记录。

### 11.3 复制本周完成

复制规则按父项去重：

- 父代办在本周完成：父项只出现一次；属于该父项且当前仍有效的子完成可以早于本周，并作为缩进明细。
- 父代办尚未完成：只有发生在本周查询区间内的子完成，才以 `父代办 / 子代办` 形式进入完成列表。
- 本周前子完成只有在同一父项的 `completed` 事件位于本周时才允许进入结果；其他父项本周完成不能为它放宽区间。
- 已撤销完成的子代办不输出。
- 父项和子项不能被重复平铺成相同层级的多条“完整任务”。

示例：

```markdown
## 本周完成

1. 写周报
   - 汇总本周完成
   - 整理本周问题
   - 写下周计划
2. 自动更新测试 / 核对安装后版本
```

## 12. 期限

第一版期限只属于顶层代办：

- 子代办编辑态不显示日期控件。
- 胶囊显示父代办期限。
- 子代办不继承、覆盖或推导独立期限。
- 父期限仍不自动排序、提醒、通知、同步日历或产生奖惩。

选择这个边界是为了避免第一版就引入以下额外问题：

- 子期限是否必须早于父期限。
- 修改父期限是否批量修改子期限。
- 胶囊应该显示父期限还是当前子期限。
- 多个子期限是否改变子项顺序。

如果真实试用证明子期限必要，后续可以复用现有可选日期控件，但仍应默认无期限、常态无占位且不自动继承。

## 13. 键盘、焦点与无障碍

| 操作 | 键盘 |
|---|---|
| 展开 / 收起子列表 | 进度按钮上的 `→` / `←`，或 `Enter` / `Space` |
| 修改标题 | 标题上的 `Enter` / `F2` |
| 保存 | 编辑框中的 `Enter` 或移开焦点 |
| 取消编辑或新增 | `Esc` |
| 完成 / 撤销子代办 | 子代办圆圈上的 `Space` |
| 同级上移 / 下移 | 排序柄上的 `Alt + ↑/↓` |
| 搜索 | `Ctrl + F` |

无障碍要求：

- 进度按钮提供 `aria-expanded` 和 `aria-controls`。
- 子列表使用嵌套 `<ol aria-label="写周报的子代办">`。
- 父行和子行使用不同选择器，防止现有顶层排序逻辑误收集子代办 ID。
- 操作结果由现有实时状态区播报，例如“已在写周报中添加子代办”。
- 完成或删除后，焦点优先落到下一同级项；没有下一项时回到上一项或父项。
- 输入法组合期间不处理保存、取消和搜索退出快捷键。

## 14. 领域模型

### 14.1 `Task` 增量字段

```rust
pub struct Task {
    // 现有字段……
    pub parent_task_id: Option<String>,
    pub sibling_position: Option<i64>,
}
```

不变式：

| 形态 | `parentTaskId` | `queuePosition` | `siblingPosition` |
|---|---:|---:|---:|
| 顶层代办 | `null` | 待办状态时必须有值 | `null` |
| 子代办 | 父代办 ID | 始终为 `null` | 必须为正整数 |

- 子代办的父项必须存在且是顶层代办。
- 禁止自引用、循环和孙级。
- 父子关系第一版创建后不可变。
- 完成子代办仍保留 `siblingPosition`。
- 已软删除子代办不进入进度分母，但投影和事件仍保留。

### 14.2 快照

保持旧字段语义并做加法：

```text
LedgerSnapshot
├─ currentTask    顶层队列第一项，语义不变
├─ queue          只包含顶层待办，语义不变
├─ completed      只包含已完成顶层待办，语义不变
└─ subtasks       新增；包含活动父项下 pending / completed 子代办
```

前端只读选择器根据 `parentTaskId + siblingPosition` 分组，并统一派生：

- 父项进度。
- 展开列表。
- 当前动作。
- 搜索上下文。

视图和控制器不得自行维护第二份子代办事实。

### 14.3 命令

新增：

- `create_subtask({ operationId, parentTaskId, title })`
- `reorder_subtasks({ operationId, parentTaskId, movedTaskId, expectedTaskIds, orderedTaskIds })`

复用并在领域层识别任务形态：

- `update_task_title`
- `complete_task`
- `undo_completion`
- `delete_task`

第一版不把 `capture_task` 改成可选 `parentTaskId`，避免改变已有操作箱载荷和快速记录语义。

### 14.4 事件与奖励

新增明确事件类型：

- `subtask_created`
- `subtask_completed`
- `subtask_completion_undone`
- `subtasks_reordered`

标题修改和软删除可以继续复用：

- `title_updated`
- `abandoned`

子代办事件的 `metadata` 必须包含父 ID 和父标题快照。子代办完成事件没有奖励交易；只有顶层 `completed` 事件产生 `+1`。

父项级联完成使用 `LedgerMutation.companionMutations` 承载子投影与子事件，但仍只调用一次 `SqliteLedgerStore::commit_transition`。伴随子事件使用内部 `commandId=cascade/<eventId>`，保存 `cascadeCommandId` 与 `cascadeParentEventId`；父完成事件保存顺序一致的 `cascadeSubtaskEventIds`，并继续使用外部 `operationId`。父事件最后插入，唯一命令回执和奖励只指向父事件。

### 14.5 父项删除不做级联写入

父项被软删除时，子代办保持原状态和关系，只从活动快照隐藏。这样可以继续使用当前单实体 `LedgerMutation`：

```text
一个 TaskWrite
+ 一个父项 abandoned 事件
+ 一份命令回执
+ 零奖励
```

禁止由前端循环调用多个 `delete_task`，也不把旧单任务删除当成失败后的静默回退。

## 15. SQLite v5 与完整性

需要把 schema 从 v4 升到 v5：

1. 迁移前继续生成并校验备份。
2. `tasks` 增加 `parent_task_id` 自外键和 `sibling_position`。
3. 旧数据全部迁移为顶层代办，两个新字段均为 `NULL`。
4. 增加 `(parent_task_id, sibling_position)` 唯一索引。
5. 全局队列和 `currentTask` 查询显式限定 `parent_task_id IS NULL`。
6. 事件表约束加入子代办事件类型。
7. 完整性报告增加层级检查：
   - 孤儿父 ID。
   - 自引用、孙级或循环。
   - 重复或不连续的同级位置。
   - 顶层与子代办队列字段形态错误。
   - 已完成父项仍存在未完成有效子代办。
   - 子代办完成事件错误地产生奖励。
   - 父标题快照和投影关系无法回放。
   - 级联子事件的内部命令格式、父子双向索引、事件时序或父主回执不一致。
   - `cascadeSubtaskEventIds` 保留键存在但类型错误、为空、重复或与实际伴随事件不一致。
8. v4→v5 必须覆盖旧回执重放、旧完成撤销、迁移备份、重新打开以及迁移后的父项级联完成。
9. schema v5 是 `0.1.2` 的正式格式；正式数据迁移到 v5 后不支持降级给 `0.1.1` 的 v4 客户端读取。安装程序回退不等于数据回退，恢复旧版必须使用迁移前受检备份。

## 16. 并发、幂等与恢复

关键并发结果必须由同一 SQLite `BEGIN IMMEDIATE` 事务内再次校验：

| 并发场景 | 合法结果 |
|---|---|
| 创建子代办 vs 完成父代办 | 完成事务始终校验全部活动子项，连预期空集合也校验；任一方先提交后，另一方成功纳入新事实或显式并发冲突，不能留下已完成父项下的未完成子项 |
| 完成最后一个子代办 vs 完成父代办 | 两次写入按事务串行；陈旧的子项版本或状态会显式冲突，重试后父项只完成仍为 pending 的子项 |
| 子代办重排 vs 删除某子代办 | 完整期望 ID 集不一致时重排失败并刷新真实快照 |
| 撤销子完成 vs 完成父项 | 不能留下完成父项和未完成子项并存 |
| 删除父项 vs 写入子项 | 删除确认后不能再向已非待办的父项添加或修改子项 |

所有新增写命令继续遵守现有操作箱规则：

- 跨 IPC 前先持久化稳定 `operationId`。
- 结果未知时保留同一 ID，禁止后续写入。
- 确认成功并刷新真实快照后才能清理操作箱。
- 损坏载荷显式报错并锁住写入，不自动删除。

顶层完成始终携带 `SubtasksUnchanged` 前置条件，比较全部活动子项的 `id / version / siblingPosition / status`，即使读取结果为空也提交 `expected=[]`。同一外部 `operationId` 丢失确认后只重放父主回执，不重复子事件或父奖励；首个伴随任务写入、首个伴随事件、主事件、奖励和提交前的故障注入均必须整事务回滚。

## 17. 前端模块落点

当前本地实现已经新增独立 `SubtaskController`，没有把所有层级操作继续塞入现有 `TaskListController`。它只维护展开、单次添加草稿、子项编辑、组内重排和焦点等临时交互状态；任务事实仍只来自后端快照。

```text
desktop/app.js                     组合根，注入所有意图回调
desktop/app/subtask-controller.js  新增、展开、子项编辑、同组重排与焦点
desktop/app/task-list-controller.js 保留顶层修改和顶层重排
desktop/app/selectors.js           只读分组、进度、当前动作和搜索投影
desktop/app/views.js               渲染父行、嵌套子列表和分组历史
desktop/app/ledger-session.js      统一可靠写入状态，不保存第二份业务数组
desktop/app/infrastructure/*       IPC 映射与操作箱 payload 校验
```

子行必须使用 `.subtask-row` / `.subtask-handle`，不能复用现有顶层 `.task-row` / `.drag-handle` 查询范围，否则顶层重排可能把子项 ID 误提交为完整主队列。

## 20. 原型说明

交互原型覆盖：

- 常态与展开。
- 首次、再次和单次添加态。
- 父标题、期限和子标题修改。
- 子代办完成、撤销和父项级联完成。
- 单个子代办删除和任务组删除确认。
- 同级键盘排序和拖动边界。
- 父子上下文搜索。
- 默认折叠、可按需展开的分组完成记录。
- 桌面胶囊当前动作。

原型使用浏览器内存数据，不连接 Tauri、SQLite、真实操作箱或正式用户数据。它用于评审交互，不代表功能已经实现。
