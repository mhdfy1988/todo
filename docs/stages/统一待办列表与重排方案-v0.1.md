# 代办：统一待办列表与重排方案 v0.1

> 状态：实现契约已冻结  
> 日期：2026-07-18

## 目标

让待办回到普通列表的使用方式：每一条都能直接勾选完成，列表顺序可以调整，第一条自然成为胶囊显示的下一件。不再把队首任务做成一张特殊卡片或单独提供文字“完成”按钮。

## 界面流程

```text
快速记录
→ 进入统一待办列表末尾
→ 任意一行可勾选完成
→ 拖动排序柄或按 Alt + ↑/↓ 调整顺序
→ 第一行自动成为胶囊中的下一件
→ 完成记录仍在独立页面查看和撤销
```

每行只包含原生勾选框、任务标题和排序柄。第一行只用细暖色线与略高字重表达，不增加“当前”“下一件”等提示文案。空列表只显示“还没有待办”。

## 写命令契约

### 任意完成

- 继续使用 `complete_task({ taskId, operationId })`。
- `taskId` 可以指向任意立即可执行的待办，不再要求位于队首。
- 任务版本、完成事件、金币奖励和命令回执仍在同一事务提交。

### 调整顺序

- 新增 `reorder_tasks({ movedTaskId, expectedTaskIds, orderedTaskIds, operationId })`。
- `expectedTaskIds` 是拖动前的完整顺序，必须与服务端当前队列完全一致，避免旧页面覆盖较新的重排。
- `orderedTaskIds` 必须是 `expectedTaskIds` 的无重复完整排列，`movedTaskId` 必须真实改变位置。
- 服务端在 `BEGIN IMMEDIATE` 事务内再次校验顺序、集合、位置和任务版本，再一次性写入连续顺序。
- 同一事务追加 `queue_reordered` 事件；事件元数据保存 `movedTaskId`、`beforeTaskIds` 和 `afterTaskIds`，用于完整性回放。
- 结果未知时继续保留同一个 `operationId`，不得只改前端 DOM 或静默回退。

## 状态变化

### 第 1 轮：完成非首项

队列为 `A / B / C`，勾选 `C`：

```text
A / B / C
→ C 完成并进入完成记录
→ 队列保持 A / B
→ 胶囊继续显示 A
```

### 第 2 轮：调整顺序

队列为 `A / B`，把 `B` 拖到 `A` 前：

```text
A / B
→ 提交 movedTaskId=B、expectedTaskIds=[A, B]、orderedTaskIds=[B, A]
→ 同事务写入 B=1、A=2 与 queue_reordered 事件
→ 快照返回 B / A
→ 胶囊显示 B
```

### 第 N 轮：并发或结果未知

- 排序期间若任务集合或版本已经变化，返回明确并发错误并刷新真实快照。
- IPC 结果未知时操作箱保留完整 ID 顺序和稳定操作 ID，重启后重放同一命令。
- 只有命令确认且真实快照刷新成功，才清理操作箱并恢复写入。

## 数据升级

- 任务表已经有唯一 `queue_position`，不新增第二套排序事实。
- schema v2 只扩展 `task_events` 允许 `queue_reordered`，并补真正的 v1→v2 增量迁移。
- 文件账本迁移前使用 SQLite `VACUUM INTO` 生成 v1 一致性备份；内存冒烟不生成文件备份。
- 迁移按“新表 → 复制 → 删除旧表 → 新表改为正式名 → 重建索引 → 外键检查”执行。

## 不变式

1. `snapshot.queue` 的顺序是唯一产品顺序，`currentTask` 必须等于它的第一项。
2. 任意完成只改变目标任务，不擅自重排其他待办。
3. 重排必须提交全部当前待办 ID，不能逐行写入或产生中间重复位置。
4. 一次重排只有一条领域事件、一个命令回执和一个事务。
5. 视图和拖动控制器只提交意图；账本事实仍由 Rust 与 SQLite 持有。
6. 完成、重排和撤销都继续使用同一操作箱恢复协议。

## 本轮范围

第一版实现任意完成、鼠标拖动、键盘重排、迁移备份、事件回放和真实桌面验证。编辑标题、阻塞/恢复、放弃/重新打开、跨状态混合排序仍留在后续阶段。
