# 代办：桌面悬浮待办产品设计方案 v0.1

> 文档状态：产品基线；统一待办列表、顺序调整、未完成标题与可选期限修改、行内删除已进入可试用状态
> 更新时间：2026-07-22  
> 产品名称：代办  
> 当前阶段：优先试用待办手感；可选期限已进入源码与自动验证，延期、阻塞、真正的“不再做”、重新打开、完整周报与通用备份/导出继续后做；宠物、AI 和日历暂停

任务状态、事件历史和金币事务的实现基线见[《任务状态与事件账本 v0.1》](./任务状态与事件账本-v0.1.md)。

## 1. 产品结论

“代办”是一款本地优先的桌面悬浮待办工具，当前先把“快速记录、统一列表、修改录错标题、按需设置期限、逐项完成、手动排序、行内删除、保留完成记录”做顺手。金币、喂食和宠物成长是暂停的后续方向，不进入当前待办试用范围。

视觉方向已经确认：三种设计全部保留，极简原生风作为第一版实现与验收基线；深色透明和像素桌宠作为后续候选皮肤，不在第一版并行开发。

一句话定位：

> 平时把事情都记下来，自己排好顺序，做完哪一件就勾掉哪一件。

当前已实现的主循环：

```text
想到事情
→ 立即记下
→ 进入统一待办列表
→ 鼠标拖动或 Alt+↑ / Alt+↓ 调整顺序
→ 第一项显示在桌面胶囊
→ 标题录错时直接在原行修改，并保留修改历史
→ 有明确期限时按需设置一个日历日，无期限时不增加界面负担
→ 任意一项完成后直接勾选
→ 保留完成记录
→ 录错的任务可点行内 × 删除，不混入完成记录
```

延期、阻塞、真正的“不再做”、重新打开、回收站/永久清理、完整周报和通用自动备份/导出属于后续待办；本地可选期限不包含提醒、通知或日历联动，宠物、AI 与日历保持暂停。

## 2. 已确认的用户习惯

- 只要想到有事情要做，就习惯先记录下来，避免遗忘。
- 通常按照列表顺序依次完成，不希望系统擅自重排。
- 部分任务暂时做不了，需要延后或等待外部条件。
- 部分任务最终确认不再做，但它们不应像删除一样消失。
- 录错或不需要保留在列表中的任务，需要能直接删除；删除不能伪装成完成。
- 完成历史是写周报和进行阶段回顾的重要依据。
- 番茄钟和专注计时暂时没有必要。
- 希望完成现实任务后获得金币，用于喂食并陪桌面宠物成长。
- 新应用将彻底取代有道待办，不长期维护两个任务入口。

## 3. 产品目标与非目标

### 3.1 当前可试用能力

1. 几秒内记下一件事情，标题之外没有必填字段。
2. 展开态使用一张统一待办列表，不再拆成“当前任务”和“接下来”。
3. 每一行都可以直接勾选，能够先完成列表中任意一项。
4. 支持鼠标拖动，也支持 `Alt+↑` / `Alt+↓` 调整聚焦项顺序。
5. 列表第一项就是桌面胶囊显示的下一件事，不额外维护“当前任务”；有任务时完成圆圈位于标题最前，已有期限固定在同一行末尾；列表为空时胶囊显示“暂无待办”并隐藏完成圆圈和期限。
6. 未完成待办的标题可在原行修改：双击标题或聚焦后按 `Enter` / `F2` 进入，`Enter` 或失焦保存，`Esc` 取消。
7. 原位编辑激活后才显示日期控件；默认值为无期限，期限使用 `YYYY-MM-DD` 日历日，清空控件即恢复无期限。
8. 无期限任务在常态列表和胶囊中不显示日期；已有期限才显示紧凑标签。任务行标签可点击编辑；胶囊期限固定在标题行末尾，是独立的 `edit-current-deadline` 按钮，点击一次即展开并聚焦当前任务日期输入。
9. 标题与期限修改分别保留前后历史，不改变任务状态、队列顺序或金币；空白标题、未变化标题和未变化期限不提交。
10. 每一行都有 `×`，可一键软删除录错或无需保留在列表中的待办。
11. 删除保留内部历史，不发金币，也不进入完成记录。
12. 顺序、标题和期限写入本地账本，应用重启后仍然保持；完成、撤销完成和删除不会清除任务原有期限字段。
13. 本地数据结构升级前先生成迁移前备份。

### 3.2 当前未纳入

- 延期、阻塞、真正的“不再做”及其重新打开继续后做；它们不是当前行内删除的同义操作。
- 回收站、删除后的用户可见恢复与永久清理继续后做。
- 完整过程历史、日期筛选和可复制周报继续后做。
- 通用自动备份、用户主动导出和安全恢复继续后做；迁移前备份不能替代这些能力。
- 宠物、金币养成、AI 和日历暂停。
- 不做有道待办同步、授权、适配、导入或历史迁移。
- 不做番茄钟或工作时长统计。
- 不做团队协作、任务指派和复杂项目管理。
- 不做算法自动排序和自动排期；设置期限不会自动重排、隐藏任务或改变胶囊第一项。
- 不做期限提醒、系统通知或日历同步；逾期与午夜标签刷新只属于前端展示。
- 不做排行榜、充值、多货币和复杂商城。
- 不做连续签到清零、宠物死亡或成长倒退。
- 不做账号系统和多设备云同步。

## 4. 核心产品原则

### 4.1 记录优先

输入一句话并回车即可保存。期限、项目、标签和备注均为可选信息，不能阻塞记录；新任务默认 `deadlineOn=null`。

### 4.2 顺序优先，但不锁死

新任务默认追加到统一列表末尾，系统不根据标题、优先级或 AI 判断擅自重排。用户可以用鼠标拖动或 `Alt+↑` / `Alt+↓` 手动调整顺序；列表第一项自然成为胶囊里的下一件事，不再单独指定“当前任务”。

### 4.3 做不了也是有效结果（后续）

延后、阻塞和放弃不是失败，不扣金币，也不触发宠物负面反馈。它们必须保留原因并进入历史。

### 4.4 宠物是伙伴，不是监工（暂停）

宠物可以庆祝、回应和成长，但不持续闪烁、不频繁说话、不责备用户，也不遮挡当前工作。

### 4.5 本地任务库是唯一事实来源

任务、顺序和完成记录全部由本地数据决定。后续状态、历史、金币和宠物成长也必须延续这一边界；未来的 AI 与日历只能提供建议或背景信息，不能静默改变任务事实。

### 4.6 历史只追加，不静默抹除

任务可以被纠正、重开或撤销完成，但已经发生的过程通过后续事件修正，不直接删除旧事件。

### 4.7 标题修改不是覆盖历史

当前标题修改只接受统一列表中的立即待办。界面更新任务当前标题的同时，账本追加 `title_updated` 事件并记录修改前后的标题；稳定任务 ID、状态、顺序和金币均不变化。已完成记录、软删除任务和其他非立即待办不提供标题修改。

### 4.8 截止日期不是“稍后再做”

`deadlineOn` 是可空的 `YYYY-MM-DD` 日历日，只回答“希望哪天前完成”；`deferUntil` 回答“何时重新进入可执行队列”。设置、修改或清除期限只追加 `deadline_updated` 历史，不自动排序、隐藏、提醒、通知、同步日历或产生奖惩。逾期状态和“今天/明天/M/D/跨年 YYYY/M/D/逾期 N 天”标签由前端按当前本地日期派生，午夜刷新不形成领域事件。

### 4.9 删除不是完成

当前行内 `×` 是一键软删除：界面立即移出待办，但内部以 `abandoned` 状态和 `metadata.action=delete` 记录，不抹除过程。它不发金币、不进入完成记录，也不代替后续带原因的“真正不再做”。

## 5. 产品结构

当前试用版包含五个主要入口：

| 入口 | 核心职责 | 桌面长期可见 |
|---|---|---:|
| 下一件任务胶囊 | 完成圆圈置于标题最前，标题可截断，已有期限固定在同一行末尾；空列表显示“暂无待办”；可展开列表或通过右上角窗口级向下箭头直接隐藏到托盘 | 是 |
| 快速记录 | 一句话新增待办 | 临时出现 |
| 统一待办列表 | 逐项勾选、行内修改标题、行内删除、鼠标拖动与键盘重排 | 展开后可见 |
| 完成记录 | 标题可截断、完成时间固定在行末；通过右侧 `↶` 图标查看并撤销已完成事项 | 否 |
| 当前页面搜索 | `Ctrl+F` 或更多菜单临时切换为搜索栏；待办与完成记录分别只按当前页标题筛选并高亮 | 否 |
| 系统托盘与设置 | 打开完整面板、临时隐藏、开机启动和退出 | 托盘常驻 |

桌面悬浮层只承担“下一件是什么”和“快速记下来”。展开后直接看到一张统一列表，不再重复展示“当前任务”和“接下来”。从系统托盘选择打开时，始终进入统一展开的完整面板，不恢复成夹在胶囊与完整面板之间的半展开窗口。完整周报、设置与暂停中的宠物养成后续放在独立页面，避免桌面长期拥挤。桌面层的“收起”仍保留任务胶囊；胶囊右上角的窗口级向下箭头可直接“隐藏”并只留系统托盘，不需要先展开，且与任务删除操作分离；“退出”结束整个应用。整窗鼠标穿透已经移除：用户看不到被窗口遮挡的内容时，这个开关没有清晰用途，也会使当前任务无法操作。

## 6. 核心使用流程

### 第 1 轮：随手记录

用户想到“更新部署说明文档”：

1. 展开面板并聚焦快速记录输入框。
2. 输入任务标题。
3. 回车保存。
4. 新任务追加到队尾。
5. 当前正在处理的任务保持不变。

### 第 2 轮：完成任意一项

列表顺序是“整理巡检报告、更新部署说明、回复项目反馈”，但用户先完成了第三项：

1. 直接勾选“回复项目反馈”这一行。
2. 写入完成事件和完成时间。
3. 该项移入完成记录，其余待办保持原顺序。
4. 如果勾选的是第一项，新的第一项自动成为胶囊里的下一件事。

### 第 3 轮：调整顺序

用户希望先做“更新部署说明”：

1. 用鼠标把该行拖到列表第一项；也可以聚焦该行后按 `Alt+↑`。
2. 松开鼠标或完成键盘移动后，一次性保存新顺序。
3. 胶囊立即显示新的第一项。
4. 关闭并重新打开应用，顺序保持不变。

### 第 4 轮：纠正录错的标题

用户把“更新部署说明”误写成了“更新部署说名”：

1. 双击未完成待办的标题；也可以先聚焦标题，再按 `Enter` 或 `F2`。
2. 在原行改成“更新部署说明”，按 `Enter` 或让输入框失焦后保存；输入法仍在组词时不会提前提交。
3. 如果按 `Esc`，恢复原题且不收起窗口；空白标题和未变化标题也不提交。
4. 保存后追加标题修改事件，任务仍在原位置，不改变状态和金币。

### 第 5 轮：删除录错的任务

用户误记了一条“服务”：

1. 点击这一行右侧的 `×`。
2. 任务立即移出统一待办列表。
3. 内部以软删除事件保留过程，不产生金币，也不进入完成记录。
4. 当前版本不提供回收站入口、删除恢复或永久清理；这些能力继续后做。

### 后续流程：任务被卡住

“测试登录功能”缺少测试环境权限：

1. 选择“暂时做不了”。
2. 选择“被卡住”。
3. 填写“等待测试环境权限”。
4. 任务进入阻塞区，并移出可执行列表。
5. 系统切换下一条可执行任务。
6. 本周回顾将其归入“阻塞事项”。

权限到位后，用户选择“恢复处理”，任务重新进入待处理队列。

### 后续流程：任务延后

1. 选择“稍后再做”。
2. 选择明天、下周或具体时间。
3. 任务保留 `pending` 状态，但在恢复时间前不进入可执行列表。
4. 到期后恢复可执行，不强制打断正在处理的任务。

### 后续流程：确认不再处理

1. 选择“不再做”。
2. 填写计划调整原因。
3. 任务进入已放弃状态。
4. 不发金币，也不扣金币。
5. 本周回顾将其归入“计划调整”。

“不再做”用于表达计划变化，需要原因并进入周报；它与当前用于清理录错内容的行内 `×` 不同。重新打开“不再做”的任务也属于后续状态闭环。

### 后续流程：周末回顾

1. 打开“本周回顾”。
2. 查看本周完成、进行中、阻塞和计划调整。
3. 取消不需要写入周报的项目。
4. 复制 Markdown 或纯文本草稿。
5. 在其他文档中继续编辑。

## 7. 任务状态模型

### 7.1 主状态

| 状态 | 中文含义 | 可执行 | 终态 |
|---|---|---:|---:|
| `pending` | 待处理 | 未延后时可执行 | 否 |
| `blocked` | 被阻塞 | 否 | 否 |
| `completed` | 已完成 | 否 | 是 |
| `abandoned` | 已放弃 | 否 | 是 |

“延后”是一项带时间的操作，不是独立终态。任务仍为 `pending`，通过 `deferUntil` 暂时退出可执行队列。

“截止日期”同样不是独立状态，但它与延后完全独立：`deadlineOn` 不决定任务是否可执行，也不改变 `pending` 队列位置。任务可以没有期限；后续即使加入延后能力，也允许延期时间与截止日期分别存在。

当前软删除复用 `abandoned` 终态，但通过 `metadata.action=delete` 明确标记删除语义。后续“真正不再做”也可以进入 `abandoned`，但必须使用不同动作语义并保留原因，不能与删除混为一谈。

### 7.2 主要状态变化

```text
待处理 ─────────→ 已完成
   │                 │
   ├──→ 被阻塞 ──────┤
   │       │         │
   └──→ 已放弃       └──→ 待处理（撤销完成 / 重开）
           │
           └────────────→ 待处理（重新打开）
```

### 7.3 不变式

- 统一待办列表只有一套顺序事实；第一项就是胶囊中的下一件事。
- 当前任意待办都可以被勾选完成，不要求先成为第一项。
- 搜索只产生当前页面的临时过滤投影；筛选时保留完成、修改、删除与撤销，但禁用重排，不新增第二套队列事实。
- 顺序调整必须整体提交，失败时不能只保存一部分位置。
- 完成、阻塞和放弃的任务不能留在可执行列表中。
- 软删除任务不出现在完成记录中，但对应任务和事件必须保留在本地历史中。
- 每个任务的有效完成奖励净值最多为一次。
- 撤销完成时同步撤销对应奖励；再次完成只能恢复净奖励，不能重复发币。
- 金币余额由奖励账本汇总，不维护一个没有来源的可变余额。
- 延后、阻塞、放弃和删除不产生奖励，也不扣金币。
- 设置、修改、清除或逾期均不产生奖励、不扣金币，也不触发自动任务变化。
- 宠物不会因用户长期未使用而死亡、退化或丢失成长值。
- AI、日历和后台任务均不得静默改变任务顺序或胶囊中的第一项。

## 8. 奖励与宠物成长（暂停方案）

本节只保留未来设计，不属于当前可试用能力，也不阻断待办主功能。

### 8.1 候选经济规则

| 行为 | 金币 | 成长值 |
|---|---:|---:|
| 一项任务首次有效完成 | +1 | 0 |
| 撤销该次完成 | -1 | 0 |
| 喂食一次 | -3 | +1 |
| 延后 / 阻塞 / 放弃 | 0 | 0 |

第一版统一奖励，不根据“重要”“紧急”或标题长度设置倍率，避免为了金币制造无意义的小任务。

### 8.2 成长阶段

| 成长值 | 阶段 |
|---:|---|
| 0～4 | 幼崽 |
| 5～14 | 少年伙伴 |
| 15 及以上 | 成熟伙伴 |

阈值属于试用参数，可调整；已获得成长值不得因阈值调整而丢失。

### 8.3 候选宠物反馈

- 待机呼吸与轻微动作。
- 完成任务后开心或捡到金币。
- 喂食反馈。
- 跨成长阶段反馈。
- 点击宠物查看今日完成数、金币和成长进度。

动画必须短暂，不能阻塞任务操作。连续完成多项时合并反馈，避免反复打断。

## 9. 历史与周报

当前用户可见的历史只有完成记录及撤销入口；软删除事件已经保留在内部账本，但还没有回收站或恢复入口。以下完整事件查询、日期筛选和周报输出均为后续方案。

### 9.1 保存的事件

- 创建、改名和调整顺序。
- 调整统一列表顺序。
- 延后及恢复时间。
- 阻塞及原因。
- 恢复处理。
- 完成及撤销完成。
- 放弃及重新打开。
- 删除事件已保留；回收站恢复和永久清理继续后做。

### 9.2 非 AI 周报模板

```markdown
## 本周完成

1. 完成月度巡检报告
2. 更新部署说明文档

## 进行中

1. 桌面宠物待办设计

## 阻塞事项

1. 登录功能测试：等待测试环境权限

## 计划调整

1. 旧方案调研：已确认改用官方 SDK
2. 数据整理：延后至下周
```

支持按自然周或自定义日期统计、选择是否纳入单项、复制 Markdown 或纯文本。周报是可编辑草稿，原始历史始终保持不变。

## 10. 取代有道的边界

### 10.1 边界

- 新应用启用后成为唯一日常待办入口。
- 不读取、不写入、不同步有道待办，也不提供有道导入或历史迁移能力。
- 旧内容需要时仍在原应用中查看；新应用不会声明已经接收或转换这些数据。
- 新应用只统计本地启用日期之后产生的完整历史。

## 11. 本地数据与可靠性

### 11.1 核心实体

`Task`：任务当前状态与展示信息。

| 关键字段 | 含义 |
|---|---|
| `id` | 本地稳定标识 |
| `title` | 任务标题 |
| `status` | 待处理、阻塞、完成、放弃 |
| `order` | 队列顺序 |
| `createdAt` / `updatedAt` | 创建与更新时间 |
| `completedAt` | 当前有效完成时间 |
| `deferUntil` | 延后至何时；决定何时重新进入可执行队列 |
| `blockReason` / `abandonReason` | 当前原因 |
| `projectId` / `tags` | 可选归类 |
| `deadlineOn` | 可选截止日；`null` 或 `YYYY-MM-DD`，不承载时刻与时区 |

`TaskEvent`：不可静默覆盖的状态变化记录。

`RewardTransaction`：完成奖励、奖励撤销和喂食支出的独立账本；宠物恢复后再启用对应产品入口。

`PetProfile`：宠物名称、种类、成长值、阶段和最近喂食时间；当前暂停。

### 11.2 当前可靠性

- 任务与统一列表顺序写入本地账本，应用重启后保持一致。
- 未完成待办的标题修改写入 `title_updated` 事件，保存 `beforeTitle` / `afterTitle`，不改变队列和金币。
- 待办期限修改通过 `update_task_deadline({ operationId, taskId, deadlineOn })` 写入 `deadline_updated` 事件，保存 `beforeDeadlineOn` / `afterDeadlineOn`；`null` 表示无期限或清除期限，同值修改显式拒绝。
- 行内删除以软删除状态和动作元数据写入账本，不产生完成奖励，也不进入完成记录。
- 当前源码数据库结构为 v4；v1、v2、v3 按级迁移到 v4，磁盘旧库升级前先生成并验证 before-v4 迁移前备份。
- 所有任务事件和奖励交易具有稳定标识，重复执行不会重复生效。
- 当前统一门禁已通过 102 个前端测试、48 个 Rust 测试，独立账本冒烟和桌面联合冒烟均为 `passed=true`；前端门禁覆盖待办/完成记录搜索、输入法组合态、安全高亮、焦点恢复与搜索态重排锁；联合冒烟包含 `updatedPendingTaskDeadline=true` 和删除后事件数 9。

### 11.3 后续可靠性

- 异常退出后的完整恢复继续扩充到全部后续状态。
- 通用自动本地备份、用户主动导出和安全恢复继续后做。
- 恢复失败时不得覆盖当前有效数据。

## 12. AI 与日历的暂停边界

### 12.1 AI

AI 当前暂停。待待办主功能完成真实试用后，才重新评估以下优先级：

1. 根据真实历史生成可追溯的周报草稿。
2. 为过大的任务提供拆分建议。
3. 从自然语言中建议日期、项目和标签。
4. 提供任务顺序建议。
5. 让宠物根据真实进度生成简短陪伴文案。

AI 只能建议或生成草稿，不能自动完成、删除、延后、放弃任务，也不能计算金币或成长值。AI 不可用时，非 AI 周报和全部核心功能必须照常使用。

### 12.2 日历

日历当前暂停。已实现的 `deadlineOn` 是本地任务字段，不读取、不创建也不同步外部日历事件。未来如果恢复日历能力，仍按“日历回答什么时候发生，待办回答要做什么”的边界推进：

1. 只读显示今天和下一项日程。
2. 在“今日时间线”中并列展示任务与日程。
3. 用户主动将任务安排到一个时间段。
4. 最后才考虑带冲突规则的写入或双向联动。

外部日程不等于任务完成，会议结束不自动发金币；日历变化不能修改任务状态和队列顺序。

## 13. MVP 验收标准

### 桌面生命周期与视觉实现

桌面生命周期的首轮可运行实现与证据见[《桌面悬浮窗口技术实验 v0.1》](./桌面悬浮窗口技术实验-v0.1.md)。当前已验证单屏 125% 下的置顶、不抢焦点、窗口形态、托盘显隐和位置恢复；多显示器完整实机矩阵仍是发布前必过项。整窗鼠标穿透不再属于产品能力或验收项。

- 第一版只以极简原生风作为实现与验收对象；另外两套皮肤不是发布阻塞项。
- 展开、下一件任务胶囊、贴边露出和隐藏到托盘的语义明确；仅宠物形态随宠物功能暂停。
- 从托盘打开时始终显示统一展开的完整面板，不出现半展开窗口。
- 收起、隐藏和退出必须是三个不同动作。
- 收起态不遮挡系统托盘，展开面板不越出当前显示器。
- 悬浮窗、金币反馈和宠物动画不得抢走其他应用的键盘焦点。
- 关闭透明效果或减少动态效果后，全部任务操作仍然清楚可用。
- 多显示器、100%～200% 缩放、任务栏不同方向和重启位置恢复通过技术验证。

### 记录与队列

- 只输入标题并回车即可成功新增。
- 新任务默认进入队尾，连续新增后顺序正确。
- 未完成待办可通过双击标题或聚焦后按 `Enter` / `F2` 进入行内编辑。
- 待办与完成记录都可按 `Ctrl+F` 或从更多菜单搜索当前页标题；无命中提示明确，`Esc` /“取消”恢复完整列表。
- `Enter` 和失焦保存有效修改；`Esc` 取消并保留原题，且不会同时收起窗口；输入法组词不会被误提交。
- 空白标题、未变化标题、已完成或已删除任务均不会产生标题修改写入。
- 标题保存后任务 ID、状态、顺序和金币不变，重启后显示新标题，事件历史仍可查到修改前后的标题。
- 原位编辑激活后才出现日期控件；无期限任务在常态行和胶囊中没有日期占位。
- 期限只接受 `null` 或合法 `YYYY-MM-DD`；设置、改期和清除后均能从事件历史还原前后期限。
- 已有期限按今天、明天、月/日、跨年月/日或逾期天数紧凑显示；任务行标签可点击编辑，胶囊期限固定在标题行末尾，点击可一次完成展开并聚焦日期输入。
- 编辑期间隐藏原期限标签；日期选择或清除成功后，焦点回到任务标题。
- 期限变化和午夜刷新不改变任务顺序、可执行性、金币或事件数量，也不触发提醒、通知和日历同步。
- 每一行都能独立勾选，完成后其余待办保持原顺序。
- 每一行都能通过 `×` 一键软删除；删除保留内部历史、不发金币且不进入完成记录。
- 鼠标拖动与 `Alt+↑` / `Alt+↓` 都能调整顺序。
- 统一列表第一项与胶囊内容一致，不存在第二套“当前任务”状态。
- 重启应用后任务顺序保持一致。

### 后续状态与历史

- 完成、延后、阻塞、恢复、放弃、重开均有可查询事件。
- 阻塞与放弃必须保留原因。
- 真正的“不再做”、重新打开、回收站恢复和永久清理仍按后续能力验收。
- 误操作可以撤销，旧事件不被静默删除。
- 本周完成、进行中、阻塞和计划调整分类正确。

### 暂停的奖励与宠物

- 每项任务最多产生一份有效完成奖励。
- 撤销完成同步撤销对应金币。
- 喂食后金币、成长值和账本保持一致。
- 金币不足时不能喂食，余额不会变为负数。
- 长期不使用不会导致宠物死亡或退化。

### 数据可靠性

- 核心功能完全离线可用。
- 异常退出与重启不会丢任务或重复发币。
- 数据结构升级前先生成迁移前备份。
- 通用自动备份、用户主动导出与安全恢复仍是后续验收项。
- 应用不依赖有道、日历、AI 或其他外部服务即可完成全部核心流程。

## 14. 第一版成功判断

第一版是否成功，不以功能数量判断，而看五件事：

1. 想到事情时，是否愿意从悬浮入口立即记录。
2. 日常是否愿意在统一列表中自己排顺序、完成哪项就勾掉哪项。
3. 做不了的事情是否能明确处理，而不是无限堆积。
4. 待办核心稳定后，是否仍需要恢复金币和宠物成长。
5. 完整周报完成后，历史是否真正减少回忆和整理成本。

当前先验证前两项的待办手感；延期、阻塞、放弃、完整周报与通用自动备份/导出完成后，再评估宠物、AI 和日历是否值得恢复。
